Version: 1.0.0

Status: Draft

Owner: Chief Software Architect

Last Updated: 2026-07-03

---

# 1. Executive Summary

Business Events are the fundamental execution unit of AI Commerce OS.

Every business activity shall begin with an immutable Business Event and eventually produce one or more new Business Events.

The Runtime does not execute business logic directly.

Instead, the Runtime continuously processes Business Events through Coordinators, Workflows and Business Capabilities.

This document defines:

- Business Event model
- Event lifecycle
- Event metadata
- Execution Context
- Event Bus
- Event routing
- Reliability
- Event versioning
- Architecture constraints

---

# 2. Purpose

The purpose of this document is to establish a unified event-driven architecture for AI Commerce OS.

All Runtime Components shall communicate through Business Events.

Direct invocation between Runtime Components is prohibited.

---

# 3. Scope

This document defines:

- Business Events
- Event Bus
- Event Lifecycle
- Event Routing
- Execution Context

This document does not define:

- Business Rules
- Workflow Definitions
- Database Schema
- Platform APIs

These are defined by other architecture documents.

# 4. Event Philosophy

## 4.1 Event First

Every business process begins with an event.

Examples:

- Order Paid
- Product Created
- Inventory Changed
- Customer Message Received

Business Events represent facts.

They do not represent commands.

---

## 4.2 Event Driven Runtime

The Runtime continuously performs the following loop.

```
Receive Event

↓

Validate Event

↓

Create Execution Context

↓

Dispatch Workflow

↓

Execute Capability

↓

Publish New Event
```

The Runtime never executes business logic directly.

---

## 4.3 Immutable Event

Business Events are immutable.

Once published, an Event shall never be modified.

Changes shall always generate a new Event.

Example:

```
Order.Created

↓

Order.Paid

↓

Order.Shipped

↓

Order.Completed
```

---

# 5. Business Event Model

A Business Event represents a completed business fact.

Each Event shall contain:

- Event Type
- Event ID
- Timestamp
- Source
- Payload
- Metadata

Business Events shall be self-describing.

---

## 5.1 Event Categories

Business Events are classified into four categories.

### External Events

Generated by external commerce platforms.

Examples:

- Order.Created
- Order.Paid
- Customer.MessageReceived
- Refund.Requested

---

### Internal Events

Generated by Runtime Components.

Examples:

- Inventory.Reserved
- Workflow.Completed
- Image.Generated
- Price.Calculated

---

### Human Events

Generated by human approval.

Examples:

- Product.Approved
- Product.Rejected
- Price.Approved

---

### System Events

Generated by the Runtime.

Examples:

- Runtime.Started
- Runtime.Recovered
- Gateway.Disconnected
- Health.Degraded

---

## 5.2 Event Naming Convention

Event names shall follow:

```
Domain.Action
```

Examples:

```
Order.Created

Order.Paid

Product.Published

Inventory.Reserved

Workflow.Completed

Runtime.Started
```

Undefined naming styles are prohibited.

---

# 6. Event Lifecycle

Every Event follows the lifecycle below.

```
Created

↓

Validated

↓

Queued

↓

Dispatched

↓

Processing

↓

Completed

↓

Archived
```

Failure path:

```
Processing

↓

Failed

↓

Retrying

↓

Processing

↓

Dead Letter
```

Dead Letter Events require manual investigation.

---

# 7. Event Metadata

Every Event shall include metadata.

Required metadata:

- Event ID
- Trace ID
- Correlation ID
- Tenant ID
- Store ID
- Platform
- Source Component
- Timestamp
- Version
- Priority

Metadata shall never contain business logic.

---

# 8. Execution Context

## Definition

Execution Context is the runtime environment created for processing a Business Event.

Runtime Components shall receive Execution Context instead of raw Events.

---

## Responsibilities

Execution Context provides:

- Business Event
- Runtime Configuration
- Business Policies
- Workflow Information
- Retry Information
- Trace Information

Execution Context shall remain immutable during execution.

---

## Structure

Execution Context includes:

- Context ID
- Runtime ID
- Event
- Metadata
- Policy
- Workflow
- Operator
- Retry Count
- Trace ID
- Correlation ID

---

# 9. Event Bus

The Event Bus is responsible for transporting Business Events.

Responsibilities include:

- Publish
- Subscribe
- Dispatch
- Persistence
- Replay

The Event Bus shall remain platform independent.

---

## Publish

Any Runtime Component may publish Events.

---

## Subscribe

Components subscribe to Event Types instead of specific Components.

---

## Dispatch

Coordinator dispatches Events according to Workflow Definitions.

---

## Persistence

Business Events shall be persisted before processing.

---

## Replay

Historical Events may be replayed for recovery or auditing.

---

# 10. Event Routing

Runtime Components shall never invoke each other directly.

Routing model:

```
Business Event

↓

Coordinator

↓

Workflow

↓

Capability

↓

Business Event
```

Platform Gateways receive only published Events.

---

# 11. Event Reliability

## Retry

Transient failures shall be retried automatically.

---

## Idempotency

Every Event shall contain an Idempotency Key.

Duplicate Events shall not produce duplicate business execution.

---

## Deduplication

Duplicate Events shall be discarded safely.

---

## Dead Letter Queue

Events exceeding retry limits shall enter the Dead Letter Queue.

Manual intervention is required.

---

# 12. Event Versioning

Business Events shall support version evolution.

Backward compatibility shall be maintained whenever possible.

Breaking changes require a new Event Version.

---

# 13. Architecture Constraints

EV-001

Business Events represent facts, never commands.

---

EV-002

Business Events are immutable.

---

EV-003

Runtime Components shall communicate only through Events.

---

EV-004

Direct invocation between Runtime Components is prohibited.

---

EV-005

Execution Context shall be created before Capability execution.

---

EV-006

Events shall be persisted before processing.

---

EV-007

Business execution shall be idempotent.

---

EV-008

All Event names shall follow Domain.Action.

---

EV-009

Failed Events shall enter Retry or Dead Letter Queue.

---

EV-010

Business Events shall remain platform independent.

# Architecture Statement

Business Events are the only communication mechanism inside AI Commerce OS.

The Runtime observes Events.

The Coordinator dispatches Events.

Workflows organize Events.

Capabilities execute business logic.

Platforms publish and consume Events.

Everything in AI Commerce OS begins and ends with a Business Event.
